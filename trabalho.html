Paradigma Orientado a
Eventos

Nosso objetivo é explorar como esse paradigma facilita a criação de sistemas interativos e responsivos, onde o fluxo de execução é guiado por eventos como cliques do usuário, entradas de dados ou outras interações.

O que é


Diferente de aguardar por um comando completo que processa a informação, o sistema em tal paradigma é programado em sua base em um laço de repetição de eventos, que recebem repetidamente informação para processar e disparam uma função de resposta de acordo com o evento.

Uma curiosidade deste paradigma é que ele trabalha com o Princípio de Hollywood: "Não nos telefone, nós telefonaremos para você"

"Diferente de programas tradicionais que seguem um fluxo de controle padronizado, o controle de fluxo de programas orientados a evento são guiados por indicações externas, chamadas eventos"

- Wikipédia

Como surgiu e quem inventou

Segundo Simon Thompson, em seu livro Haskell: The Craft of Functional Programming (ou A Arte da Programação Funcional), paradigmas de programação são formas ou técnicas de representar situações do mundo real ou imaginário usando um computador. Cada paradigma fornece diferentes ferramentas para construir modelos e permite pensar nos problemas de diversas maneiras, resolvendo-os da forma mais adequada.

Programação Orientada a Eventos (POE) em C#


Em uma arquitetura orientada a eventos, os aplicativos agem como produtores de eventos ou consumidores de eventos (e muitas vezes como ambos). Um produtor de eventos transmite um evento, no formato de uma mensagem,para um broker ou outra forma de roteador de eventos em que a ordem cronológica do evento é mantida em relação a outros eventos.

Um consumidor de eventos ingere a mensagem, em tempo real (assim que elas surgem) ou em qualquer outro horário escolhido, e processa a mensagem para acionar outra ação, fluxo de trabalho ou seu próprio evento.



Exemplo de aplicação


Em um exemplo simples, um serviço bancário poderia transmitir um evento "depósito", que outro serviço no banco consumiria e responderia registrando um depósito no extrato do cliente.

Porém, as integrações orientadas a eventos também podem acionar respostas em tempo real, com base na análise complexa de grandes volumes de dados, como o "evento" quando um cliente clica em um produto de um site de ecommerce e são geradas recomendações instantâneas do produto com base nas compras de outros clientes.

Desvantagens

O paradigma de programação orientada a eventos pode apresentar algumas desvantagens, como:

2. Confiabilidade: A estabilidade depende da confiabilidade dos serviços de gerenciamento de filas e barramentos
3. Custo: O custo de implantação pode ser um desafio
4. Concorrência: Pode haver muita concorrência
5. Perda de informação: Pode haver perda de informação
6. Garantia de entrega: A garantia de entrega pode ser um problema em grandes volumes
7. Governança de dados: Pode ser um desafio para a governança de dados e fluxo

Arquitetura orientada a eventos



Tudo concentrado ao Event Broker.


É um centralizador de eventos, através do broker os eventos gerados pelos sistemas são organizados e repassados para os consumidores corretos. Quem gera o evento envia o mesmo a um broker, os consumidores recebem o evento e finalmente ocorre o processamento alimentando uma fonte de eventos dessa forma podemos posteriormente recriar todo o caminho percorrido por esse evento e até recriar a sua linha do tempo.


Vantagens


1. Quando a aplicação precisa de escalabilidade
2. Alta demanda e alta velocidade dos dados
3. Resiliência e confiabilidade
4. Disponibilidade das aplicações, podemos ter diversos consumidores em infraestruturas distintas.
5. Alta coesão e baixo acoplamento haja vista que o cliente não conhece o consumidor


Desvantagens


1. Alto nível de complexidade se não utilizado de forma correta
2. A estabilidade está diretamente ligada à confiabilidade nos serviços de gerenciamento de filas e barramentos, se esse serviço cair, toda a aplicação para
3. Custo de implantação é um desafio

Linguagem de Programação
C#

C# é uma linguagem de programação, multiparadigma, de tipagem forte, desenvolvida pela Microsoft como parte da plataforma .NET. A sua sintaxe orientada a objetos foi baseada no C++ mas inclui muitas influências de outras linguagens de programação, como Object Pascal e, principalmente, Java.

Pilares da POO



Abstração

É a capacidade de representar elementos do mundo real em um programa, focando em suas características mais relevantes e ignorando os detalhes de implementação.


Encapsulamento

É o mecanismo que protege os dados internos de um objeto, controlando o acesso a eles através de métodos.


Herança

Permite criar novas classes (classes filhas) a partir de classes existentes (classes pai), herdando seus atributos e métodos.


Polimorfismo

Capacdade de um objeto assumir múltiplas formas, ou seja, de um mesmo método se comportar de maneiras diferentes em classes diferentes.



Sintaxe e Semântica

Em C#, a sintaxe e a semântica são conceitos fundamentais para entender como o código é escrito e o que ele realmente faz.



Sintaxe

A sintaxe refere-se às regras e à estrutura do código. É como você deve escrever seu código para que o compilador o entenda. Aqui estão alguns exemplos de elementos de sintaxe em C#:

Declaração de variáveis:

                  
  int numero = 10;
  string nome = "João";

Estruturas de Controle:

                  
if (numero > 5)
{
  Console.WriteLine("Maior que 5");
}
else
{
	Console.WriteLine("Menor ou igual a 5");
}

Laços:

                  
for (int i = 0; i < 10; i++)
{
	Console.WriteLine(i);
}

Métodos:

                  
private void btnCadastrar_Click(object sender, EventArgs e)
{
  // Valida os dados e salva o cliente
  MessageBox.Show("Cliente cadastrado com sucesso!")
}


Semântica

A semântica diz respeito ao significado do código. É como a lógica e o comportamento do seu código se manifestam quando executados. Por exemplo: 

                  
namespace CompararNumeros
{
    class Program
    {
        static void Main(string[] args)
        {
            int numero;

            Console.WriteLine("Digite quantos números você quer comparar:");
            int quantidade = int.Parse(Console.ReadLine());

            for (int i = 0; i < quantidade; i++)
            {
                Console.WriteLine("Digite um número:");
                numero = int.Parse(Console.ReadLine());

                if (numero > 5)
                {
                    Console.WriteLine("O número " + numero + " é maior que 5!");
                }
                else
                {
                    Console.WriteLine("O número " + numero + " não é maior que 5.");
                }
            }
        }
    }
}

No código acima, a sintaxe pode estar correta, mas a semântica deve fazer sentido. Se você escreveu if (numero < 5), mas quer que a mensagem "Maior que 5" apareça para números maiores que 5, isso é um erro semântico.


Resumo

Sintaxe: Refere-se à forma como o código é escrito. Deve seguir as regras do C# para ser compilado corretamente.

Semântica: Refere-se ao significado do código. É importante que a lógica e a intenção do programador sejam corretamente refletidas no código.
